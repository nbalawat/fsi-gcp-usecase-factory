# archives/

Long-lived audit artifacts. Things that must be kept FOREVER for regulator
audits but should NOT clutter the per-use-case trees.

## Layout

```
archives/
└── design/                              ← /fsi-design-review writes here
    └── <use-case>/
        └── <ISO-timestamp>/
            ├── _review.html             ← side-by-side comparator from the round
            ├── option-<rejected>/       ← each loser's full directory
            │   ├── manifest.yaml        (with rejected: true + rejection_reason stamped)
            │   ├── rationale.md
            │   ├── tradeoffs.md
            │   ├── components/
            │   ├── app/
            │   └── Dockerfile
            └── option-<rejected>/
                ...
```

## What this captures (and why)

When a regulator or internal-audit reviewer asks "why did this UC end up
with THIS UX and not something else?", the answer lives here:

- The comparator HTML shows what the 4 (or 7, including re-spins) options
  looked like side-by-side
- Each rejected option's `rationale.md` explains what the agent was going for
- Each rejected option's `tradeoffs.md` explains what it optimised for
- `decision.yaml` (in the UC tree) names which option won and the user's
  keep/drop/change annotations

Together: complete provenance of a design decision.

## What this does NOT contain

- The **winning** option — the canonical promoted code lives at
  `usecases/<uc>/ui/components/+app/`. The winner's **source pin**
  (what the agent emitted before promotion) lives at
  `usecases/<uc>/ui/proposals/option-<chosen>/`. Both stay in the UC tree
  because they ARE the UC, not historical artifacts.
- Ephemeral Cloud Run URLs — those are torn down after the user picks.
  The screenshots in each option's manifest are the visual record.
- Per-build artifacts (Docker layers, GCS blobs) — handled by GCP-side
  lifecycle policies, not the repo.

## Rules

- Files in this tree are **append-only**. Nothing is ever modified.
  The architecture-auditor fails any commit that deletes or modifies an
  existing archive file.
- Subdirectories are timestamped to ISO-8601 UTC compact form
  (`20260510T142233Z`). Re-spins of the same UC produce a new timestamp.
- No human-edited files. Everything is generated by `/fsi-design-review`
  during promotion.

## When to grep this

- "Show me every UC where the user picked the wildcard option" →
  `grep -r 'variation_axis: wildcard' archives/design/*/manifest.yaml | grep -v 'rejected: true'`
- "What density level did we end up converging on?" → grep `density_score`
- "Did any UC pick a metaphor-first design?" → grep `primary_metaphor`

## How to retire (you almost never should)

Archives are intentionally permanent. If you absolutely need to retire one
(e.g. a UC was decommissioned years ago):

1. Get arch-review approval (the audit trail is the bank's protection against "we don't know why we built it this way")
2. Move the archive subtree to `archives/_retired/<uc>/` rather than deleting
3. Capture the rationale in a commit message that references the approval

The architecture-auditor's "files removed from archives" check is
intentionally strict. Bypassing it requires the override block in
`decision.yaml: iteration_override` plus an approved retirement ticket.
