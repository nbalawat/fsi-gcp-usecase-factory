# Option A — rationale

The canvas describes a commercial-credit underwriting case with four HITL gates and five agents in a hybrid-model pipeline. The moment of truth for the Credit Officer is not "what did each of the five agents do?" — it is "what's the recommendation, and what's the one fact that would flip it?" Option A is built around that 30-second scan.

The case-detail page is a single sparse column. The decision word (APPROVE) is rendered at 96px serif inside a tinted hero — the artifact IS the page. Below it sits one row of four rule verdicts and one row of three stat cards (revenue, debt, FCF — read verbatim from the spreader output, no math performed in the component). All other detail — citations, per-agent reasoning, the workflow timeline — is hidden behind a single `<details>` disclosure or compressed into the 240px right-rail of HITL gate statuses. The right-rail also carries the only navigation: "Open approval flow →". Total chrome on the page is the 56px top strip plus the right rail; everything else is the document.

The approval-flow page is one column, `max-w-2xl`, no rail. The four canvas gates (`extraction_review`, `rating_review`, `draft_review`, `final_approval`) each render with the shared `ApprovalGate` primitive — the contract demands every gate has an approval surface. Decided gates collapse to a one-line strip; the active gate is the only thing taking vertical space. Final approval is marked irrevocable, which triggers the shared component's confirm dialog. Nothing else on the page competes for attention. The Credit Officer scrolls once, decides once, signs once.

Reuse: eight components in `components_used`, three from the shared library (`ApprovalGate`, `StatusBadge`, `StatCard`) and five use-case-owned wrappers that compose them. The use-case components contain zero business logic — they render strings and numbers from `lib/data.ts`, which is itself a thin adapter over `_shared/mock-data.ts`. The only "logic" anywhere is `getGateStatuses()` folding the event stream into a per-gate state map, and the verdict→badge-kind mapping in `ruleVerdictBadgeKind` — both pure presentation.
