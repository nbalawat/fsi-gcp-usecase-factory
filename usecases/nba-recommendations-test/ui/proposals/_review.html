Option A · density

↗ open standalone
density ★★★★★ motion · minimal aff · inline-per-row metaphor · dense-queue-table persona · Relationship manager triaging next-best-action recommendations

Option A treats the queue AS the page. RMs see every pending recommendation as one dense row containing customer (CaseCard), recommended action + one-line rationale, confidence, annualised uplift, expiry urgency, and inline disposition buttons. Each row is a complete unit — Accept / Snooze / Dismiss are reversible and fire without leaving the table. The irrevocable "Send to customer" action is the ONLY one that routes the RM through a confirm step; it lands on /approval/[id] which wraps the shared ApprovalGate primitive. The detail view at /case/[id] is for "I need to drill in" — full rational

Option B treats the CUSTOMER RELATIONSHIP as the metaphor. Each card is one story: the agent narrates the recent activity timeline of the customer, what they noticed, why it matters now, and what to propose. Evidence + safety chips support the story; the disposition gate acts on the story. The RM reads stories, not tables. Self-contained build (Rule 38): NO _vendor symlinks. Option C · affordance

↗ open standalone
density ★★★★☆ motion · minimal aff · inline-per-card metaphor · queue-as-action-surface persona · Branch banker (RM) processing the daily NBA queue

Option C optimises for one thing: branch-banker throughput. The queue IS the action surface. Every recommendation card carries the four reversible buttons (Accept / Reject / Snooze / Escalate) directly beside the rationale that justifies them. Reject and Snooze open INLINE expanders — no modal, no bottom-bar. Accept flips the card to a "queued for send" pill with a single link out to /approval/[id], which is the ONLY irrevocable surface in the app. The /case/[id] route mirrors the same affordance: the same inline disposition row is the centerpiece, the audit timeline sits below.
