Score distribution histogram (20 buckets, banded approve/gray/decline) is the hero — every transaction is a sample contributing to one of those bars. The feature-firing list shows which inputs are actually doing the work, with 24h drift indicators per feature. A single optional highlight pin lands the active sample on the curve. Option D · wildcard

↗ open standalone
density ★★★★☆ motion · minimal aff · cell-as-button metaphor · feature-mcc-heatmap persona · Fraud Ops analyst monitoring the live fraud floor

Option D treats the firing population as a 2-dimensional heat surface: rows are the model's top fraud features, columns are merchant category codes, cells count the non-approve firings that lit up both axes. A click drills into the individual transactions in that cell; each transaction's detail page links its contributing features BACK to their cells, so the analyst can move fluidly between population view and case view.

reuse 9 + Population-level pattern detection — "the velocity-5m × wire/crypto cell just lit red" beats any single-row table for spotting attack waves+ Two-way navigation between population (heatmap cell) and case (event) — every feature pill on the event page deep-links to the cell that pattern belongs to+ Audit explainability — every count comes from a pure tally of events; no thresholds are computed in the UI+ Reuse — five shared primitives carry chrome and metrics; novelty is in the heatmap aggregator + grid − Per-event detail is secondary — readers who think first in "which transaction" need to use the right-rail stream− The 8×8 grid forces fixed feature and MCC axes; the model can fire on far more features than 8, so this view summarises (the firing-population view, not the raw-feature-vector view)− Approve-only firings are intentionally not counted in cell intensity; a quiet cell does NOT mean "no traffic," it means "no fraud signal"− There is no SLA-clock anchor; the real-time console pattern would normally surface a P50/P99 latency rail — that lives in a sibling option
